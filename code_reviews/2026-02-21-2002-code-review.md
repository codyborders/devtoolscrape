# Summary
Reviewed open GitHub issues `#31`-`#36` and prioritized by runtime impact. Applied fixes for all six issues and added regression tests. Previously fixed issues (`#2`-`#16`, excluding already closed) were closed with commit references.

## Priority Order Used
1. CRITICAL: `#31` startup crash when `OPENAI_API_KEY` is missing.
2. MAJOR: `#34` false scrape-success timestamp on total failure.
3. MAJOR: `#35` HN feed isolation failure on classifier exceptions.
4. MAJOR: `#36` Product Hunt query not scoped to last 24h trending.
5. MAJOR: `#32` Related tools empty for Hacker News/Show HN.
6. MINOR: `#33` tracing helper ignores `span_name` and misuses `resource`.

# Design Findings
## MAJOR: Inconsistent source abstraction in related-tool lookup
`database.get_related_startups` used exact `source = ?` while listing/count paths already used `SOURCE_REGISTRY` and `classify_source`. This duplicated logic and caused divergent behavior for dynamic Hacker News source strings.

## MAJOR: Runner state update lacked success guard
`scrape_all.main` wrote completion status unconditionally, making write-side behavior diverge from actual success semantics.

# Security Findings
No direct security vulnerabilities were identified in these issue fixes.

# Performance Findings
## MINOR: Product Hunt query scope and result cardinality
The Product Hunt API call now filters by `postedAfter` (last 24h) and sorts by `VOTES_COUNT`. This reduces stale all-time results and improves relevance of inserted/updated rows.

# Code Quality Findings
## CRITICAL: Import-time OpenAI client initialization
`ai_classifier` initialized OpenAI at import time even with missing API key, which could crash scrapers before fallback logic executed.

## MAJOR: Feed-level exception boundary too narrow
`scrape_hackernews._scrape_hn_feed` only caught request and a subset of parse exceptions; classifier/runtime exceptions escaped the boundary.

## MINOR: Observability helper contract mismatch
`trace_http_call` accepted `span_name` but ignored it, violating API expectations and reducing instrumentation consistency.

# Simplification Findings
- Consolidated related-startup matching onto existing source registry logic.
- Simplified scrape completion semantics to one explicit success guard.
- Standardized HTTP trace helper semantics to match `trace_external_call`.

# Changes Applied
- `ai_classifier.py`: lazy/safe OpenAI client init, no-key import safety, runtime client retrieval.
- `database.py`: source-family aware `get_related_startups` query using `SOURCE_REGISTRY`.
- `observability.py`: `trace_http_call` now uses `span_name` as span name and `resource` as Datadog resource.
- `scrape_all.py`: only records scrape completion when at least one scraper succeeds.
- `scrape_hackernews.py`: widened feed-level fallback to catch unexpected exceptions.
- `scrape_producthunt_api.py`: GraphQL query now uses `postedAfter` and `VOTES_COUNT` with variables.
- Tests updated/added across:

  - `tests/test_database.py`
  - `tests/test_observability.py`
  - `tests/test_scrape_all.py`
  - `tests/test_scrape_hackernews.py`
  - `tests/test_scrape_producthunt.py`
  - `tests/conftest.py`
