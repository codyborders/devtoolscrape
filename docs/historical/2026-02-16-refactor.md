## 1. Context Manager for Database Connections

**File:** `database.py`
**Lines affected:** Nearly every function in the file (16 functions)

### What Is a Context Manager?

A context manager is a Python pattern that guarantees cleanup code runs, even if an error occurs. You have probably seen it with file operations:

```python
# Without context manager -- if an error happens between open and close,
# the file stays open and leaks a file descriptor
f = open("data.txt")
data = f.read()
f.close()

# With context manager -- the file is ALWAYS closed, even if f.read() throws
with open("data.txt") as f:
    data = f.read()
```

### What Was Wrong?

Before this refactoring, every database function in `database.py` looked like this:

```python
def get_all_startups():
    conn = _connect()
    rows = conn.execute("SELECT * FROM startups").fetchall()
    conn.close()
    return [dict(row) for row in rows]
```

The problem: if `conn.execute()` raises an exception (say, a corrupted database file), `conn.close()` never runs. The connection leaks. In SQLite this is not catastrophic (the OS eventually reclaims it), but it violates a fundamental principle: **resources should always be released, even on error paths.**

### What Changed?

I added a `_db_connection()` context manager:

```python
@contextmanager
def _db_connection() -> Iterator[sqlite3.Connection]:
    """Open a database connection and ensure it is closed after use."""
    conn = _connect()
    try:
        yield conn
    finally:
        conn.close()
```

Then I replaced every manual open/close pattern with:

```python
def get_all_startups():
    with _db_connection() as conn:
        rows = conn.execute("SELECT * FROM startups").fetchall()
    return [dict(row) for row in rows]
```

Now `conn.close()` is guaranteed to run no matter what happens inside the `with` block.

### Why Not `init_db`?

The `init_db()` function has a retry loop -- it tries to connect twice in case of transient filesystem errors. The context manager pattern does not cleanly fit a retry-on-connect scenario, so `init_db` keeps its manual `try/finally` pattern. This is a pragmatic trade-off: we use the context manager everywhere it fits naturally, and document why the one exception exists.

### Functions Updated

`is_duplicate`, `save_startup`, `get_startup_by_id`, `get_related_startups`, `get_startups_by_sources`, `count_startups_by_sources`, `get_source_counts`, `get_all_startups`, `get_existing_startup_keys`, `count_all_startups`, `search_startups`, `count_search_results`, `get_startup_by_url`, `record_scrape_completion`, `get_last_scrape_time`

## 2. Type Hints

### What Are Type Hints?

Type hints tell Python (and humans reading the code) what types a function expects and returns:

```python
# Without type hints -- what does this return? A string? A dict? None?
def get_startup_by_id(startup_id):
    ...

# With type hints -- clearly returns a dict or None
def get_startup_by_id(startup_id: int) -> Optional[Dict[str, Any]]:
    ...
```

Type hints do not change how the code runs. Python ignores them at runtime. But they serve three purposes:

1. **Documentation:** A developer reading the code immediately knows what to expect.
2. **IDE support:** Your editor can autocomplete and catch type errors.
3. **Static analysis:** Tools like mypy can find bugs before you run the code.

### What Changed?

I added return type annotations to every function that was missing them, and parameter types where they were missing. Here are the key additions:

**database.py:**
- `_row_to_dict(row: sqlite3.Row) -> Dict[str, Any]`
- `init_db() -> None`
- `save_startup(startup: Dict[str, Any]) -> None`
- `get_all_startups(...) -> list[Dict[str, Any]]`
- `get_startup_by_url(url: str) -> Optional[Dict[str, Any]]`
- `record_scrape_completion(scrapers_run: Optional[str] = None) -> None`
- `get_last_scrape_time() -> Optional[str]`
- All query functions: `-> list[Dict[str, Any]]`
- Count functions: `-> int`
- `get_source_counts() -> Dict[str, int]`
- `get_existing_startup_keys() -> list[Dict[str, str]]`

**ai_classifier.py:**
- `_cache_get(...) -> Optional[Any]`
- `_cache_set(..., value: Any) -> None`
- `_call_openai(..., response_format: Optional[Dict[str, str]] = None) -> Any`

**scrape_producthunt.py:**

### Why `Optional[Dict[str, Any]]` Instead of `Dict | None`?

Python 3.10 introduced `X | None` syntax. Since this project targets Python 3.11, either works. I used `Optional[...]` from `typing` because it is the existing convention in this codebase. Consistency matters more than using the newest syntax.

### What Is a Module Docstring?

Every Python file is a module. The first string literal at the top of the file (before any imports) becomes the module's docstring. It tells anyone reading the file what this module does:

```python
"""SQLite persistence layer with FTS5 full-text search for developer tools."""

import sqlite3
...
```

### Why Does This Matter?

When you run `help(database)` in a Python shell or when a tool like Sphinx generates documentation, the module docstring is the first thing shown. Without it, a developer has to read the entire file to understand its purpose.

```python
def is_duplicate(name: str, url: str) -> bool:
```

## 5. Exception Handling

**Files:** `scrape_hackernews.py`, `scrape_github_trending.py`, `scrape_producthunt_api.py`, `scrape_all.py`, `ai_classifier.py`

### The Principle

PYTHON.md says: catch the most specific exception possible. A bare `except Exception:` is a code smell because it catches *everything* -- including bugs you would want to see immediately (like `NameError` from a typo, or `ImportError` from a missing module).

### The Nuance

Some `except Exception:` blocks are *intentionally* broad. In a scraper, if one story out of 50 fails for any reason, you want to log it and keep going -- you should not lose the other 49 stories. The key is to **document** when you are being intentionally broad.

### What Changed?

**Narrowed where safe:**

In the scraper outer catch-all blocks (the ones after `except requests.RequestException:`), the only errors that can realistically occur are data-processing errors: missing dict keys, bad type conversions, malformed data. I narrowed these to:

```python
except (KeyError, TypeError, ValueError, AttributeError):
```

This applies to:
- `scrape_hackernews.py`: Both `scrape_hackernews()` and `scrape_hackernews_show()` outer catch
- `scrape_github_trending.py`: Outer catch, classify error catch, and categorize error catch
- `scrape_producthunt_api.py`: Outer catch and token retrieval catch

For the token retrieval in `scrape_producthunt_api.py`, I narrowed from bare `except Exception:` to:

```python
except (requests.RequestException, KeyError, json.JSONDecodeError):
```

These are the three specific things that can go wrong: the HTTP request fails, the JSON response is missing the `access_token` key, or the response is not valid JSON.

**Kept broad with comments:**

Some `except Exception:` blocks are intentionally broad for resilience. I added comments explaining why:

- `scrape_hackernews.py` per-story fetch loops: `# Intentionally broad: one bad story must not kill the scrape loop`
- `scrape_all.py` scraper module loading: `# Intentionally broad: scraper modules may raise arbitrary exceptions`
- `ai_classifier.py` OpenAI calls and fallbacks: `# Intentionally broad: any batch failure falls back to per-item classification`, `# Intentionally broad: any classification failure degrades to keyword matching`, `# Intentionally broad: category failures return None rather than crashing`, `# Intentionally broad: log any API error and let tenacity decide whether to retry`

### Why This Matters

If you ever introduce a bug -- say, you misspell a variable name and get a `NameError` -- a broad `except Exception:` will silently swallow it. You will spend hours wondering why your scraper silently returns no results. With narrowed exceptions, the `NameError` propagates up and gives you a clear traceback pointing to the exact line.

## 6. SECRET_KEY Security Fix

**File:** `app_production.py`

### What Was Wrong?

The old code:

```python
app.config['SECRET_KEY'] = os.getenv('SECRET_KEY', 'dev-secret-key-change-in-production')
```

This means if someone forgets to set the `SECRET_KEY` environment variable in production, the app runs with a hardcoded, publicly known secret. Anyone who reads the source code on GitHub can use this key to forge session cookies.

### What Changed?

```python
_secret_key = os.getenv('SECRET_KEY')
if not _secret_key:
    _secret_key = secrets.token_hex(32)
    logger.warning("config.generated_secret_key", extra={"event": "config.generated_secret_key"})
app.config['SECRET_KEY'] = _secret_key
```

Now:
- If `SECRET_KEY` is set in the environment (the correct production setup), it uses that.
- If not, it generates a random 64-character hex string using Python's `secrets` module (cryptographically secure).
- It logs a warning so operators know the key was auto-generated (which means sessions will not survive restarts -- a signal that the env var should be set).

### Why `secrets.token_hex` Instead of `os.urandom`?

`secrets.token_hex(32)` produces a 64-character hex string from 32 random bytes. It is the standard library's recommended approach for generating security tokens (added in Python 3.6 specifically for this purpose). `os.urandom` would work too, but `secrets` is more explicit about its intent.

## 7. Unused Import Cleanup

**Files:** `logging_config.py`, `scrape_all.py`, `app_production.py`

### What Changed?

Removed imports that were never used:

- `logging_config.py`: Removed `import json` -- the module uses `pythonjsonlogger.jsonlogger`, not the `json` stdlib module. The string `"json"` in log format configuration is just a string literal, not a reference to the module.
- `scrape_all.py`: Removed `import sys` and `from datetime import datetime` -- neither was referenced anywhere in the file.
- `app_production.py`: Removed `get_startups_by_sources` from the database import -- the function was imported but never called.

### Why This Matters

Unused imports are not just clutter. They:
1. Mislead readers into thinking the module is used somewhere
2. Create false dependencies (if the imported module is removed from requirements, this file would break even though it does not need it)
3. Slow down module loading (marginally, but the principle matters)

## 8. Test Fix

**File:** `tests/test_scrape_producthunt.py`

### What Changed?

When I narrowed the exception handling in `scrape_producthunt_api.py`'s `get_producthunt_token()` from `except Exception:` to `except (requests.RequestException, KeyError, json.JSONDecodeError):`, one test broke.

The test `test_get_producthunt_token_failure` used a mock response that raised a bare `Exception("fail")` from `raise_for_status()`. In reality, `raise_for_status()` raises `requests.exceptions.HTTPError`, which is a subclass of `requests.RequestException`. The test was using a less realistic exception type.

I fixed the test to raise `requests.exceptions.HTTPError("fail")` instead, which is what the real `requests` library does. This makes the test more accurate and compatible with the narrowed exception handler.

I also added `import requests` to the test file since it was needed for the exception class reference.

### Why?

`pyproject.toml` is the modern standard for Python project configuration (PEP 621). It centralizes settings that would otherwise be scattered across `setup.cfg`, `pytest.ini`, `mypy.ini`, etc. The pytest configuration ensures tests always run from the correct directory with sensible defaults. The mypy configuration enables type checking with our Python version target.

## 10. Software Design Review Findings

I reviewed the codebase against the principles from "A Philosophy of Software Design" by John Ousterhout. These are observations for future work -- they were not fixed in this pass because they would change runtime behavior.

### Information Leakage: Dual Keyword Lists

`DEVTOOLS_KEYWORDS` exists in both `dev_utils.py` (15 keywords, regex word-boundary matching) and `ai_classifier.py` (31 keywords, substring matching). They serve different purposes:

- `dev_utils.py`: Used by the Product Hunt RSS scraper as the *only* classifier (no API calls)
- `ai_classifier.py`: Used as a permissive pre-filter before sending to the OpenAI API

The different matching strategies are intentional (the AI pre-filter can afford false positives). But maintaining two separate keyword lists that overlap significantly is a maintenance burden. A future refactoring could have ai_classifier import and extend the dev_utils list.

### Repetition: Duplicate Scraper Functions

`scrape_hackernews()` and `scrape_hackernews_show()` in `scrape_hackernews.py` are ~90% identical code. They differ only in:
- API endpoint URL (topstories vs showstories)
- Number of stories fetched (50 vs 30)
- Minimum score threshold (10 vs 5)
- Key prefix (empty vs "show-")
- Source string format

A parameterized `_scrape_hn_endpoint(url, limit, min_score, key_prefix, source_template)` helper would eliminate ~80 lines of duplication.

### Information Leakage: Source Key Mapping

The mapping from source keys (`"github"`, `"hackernews"`, `"producthunt"`) to database values (`"GitHub Trending"`, `"Hacker News%"`, `"Product Hunt"`) appears in three places: `database.py`'s `get_startups_by_source_key` and `count_startups_by_source_key`, and `app_production.py`'s `filter_by_source`. Adding a new source requires changes in multiple files.

## 11. Code Simplifier Findings

The code-simplifier agent identified several additional opportunities. The highest-impact ones for future consideration:

1. **Auto-populate `event` in logging infrastructure:** Every log call duplicates the message as `extra={"event": same_string}`. The `ContextFilter` or `DatadogJSONFormatter` could auto-copy `record.msg` into the event field, eliminating hundreds of redundant entries.

2. **Shared `_build_description` helper:** Three scrapers each build `[category] description` strings with slightly different logic. `scrape_hackernews.py` already has a `_build_description` helper that could be shared.

3. **Direct imports in `scrape_all.py`:** The dynamic `importlib.util` loading is over-engineered for a fixed set of three scrapers. Direct imports would be simpler and fail loudly if a module is missing.

4. **Consolidate similar routes in `app_production.py`:** `index()` and `filter_by_source()` share ~90% of their structure and could be merged.

These were documented but not implemented to keep the refactoring scope focused on PYTHON.md compliance.
