# Software Design

Apply the principles from "A Philosophy of Software Design" when writing, reviewing, or refactoring code.

## Two Modes

### Writing New Code

When designing or writing new code, apply these principles in order of impact:

1. **Think strategically.** Invest in design. Do not just get it working -- produce a clean design that also works. Spend 10-20% of effort on design improvements.

2. **Create deep modules.** Each module should provide powerful functionality behind a simple interface. A few deep methods beat many shallow ones. Ask: "Is my interface simpler than my implementation?"

3. **Hide information.** Each module encapsulates design decisions invisible through its interface. If a design decision appears in multiple modules, reorganize so it lives in exactly one.

4. **Make interfaces general-purpose.** Design interfaces general enough to support multiple uses, but keep functionality focused on current needs. Ask: "What is the simplest interface that covers all my current needs?"

5. **Different layer, different abstraction.** Adjacent layers should provide different abstractions. Avoid pass-through methods that just delegate with the same signature.

6. **Pull complexity downwards.** Handle complexity inside the module rather than pushing it to callers. A simple interface matters more than a simple implementation.

7. **Define errors out of existence.** Redefine APIs so error conditions cannot arise. Mask exceptions at low levels. Aggregate exception handling. Crash on truly unrecoverable errors.

8. **Design it twice.** Consider at least two radically different approaches before committing. Compare pros and cons. This applies to interfaces, implementations, and system decomposition.

9. **Name precisely.** Names should create a mental image of the entity's purpose. Avoid vague names (`data`, `result`, `info`, `tmp`). Use predicates for booleans (`isConnected`, not `status`).

10. **Be consistent.** Same name for same purpose everywhere. Same coding patterns. Same conventions. Consistency creates cognitive leverage.

11. **Make code obvious.** Code is obvious when a reader's first guesses about behavior are correct. Choose good names, be consistent, use whitespace, add comments only where code is non-obvious.

12. **Write comments first.** Write interface comments before implementation. If a method needs a long comment to describe, it has a complex interface -- redesign it. Comments capture information that cannot be represented in code.

### Reviewing Existing Code

When reviewing code, scan for red flags in this order:

1. **Shallow modules** -- Interface complexity comparable to implementation. Many trivial methods. Classes with minimal functionality.

2. **Information leakage** -- Same design decision reflected in multiple modules. Changing one detail requires coordinated changes across classes.

3. **Temporal decomposition** -- Code structured around time order of operations rather than information hiding. Separate read/write classes for the same format.

4. **Pass-through methods** -- Methods that do nothing except invoke another method with the same signature. 50%+ of methods just delegate.

5. **Overexposure** -- Using a common feature forces learning about rare features. API requires information that could be defaulted.

6. **Conjoined methods** -- Understanding one method requires understanding another's implementation.

7. **Repetition** -- Same non-trivial code pattern in multiple places.

8. **Vague or hard-to-pick names** -- Names broad enough to refer to many things. Difficulty finding a precise name suggests the entity tries to do too much.

9. **Comment problems** -- Comments that restate code. Implementation details in interface documentation. Entities requiring long, complicated descriptions.

10. **Special-general mixture** -- General-purpose mechanism containing code specialized for a particular use case.

11. **Nonobvious code** -- Behavior that cannot be understood with a quick reading. Surprising side effects.

12. **Too many exceptions** -- Exceptions the caller cannot handle. Error handling longer than normal-case code. Error conditions that could be eliminated by redefining the operation.

For a detailed checklist, read `references/red-flags.md`.
